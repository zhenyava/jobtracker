# PRODUCT VISION & SPECS: JOB TRACKER SAAS

## 1. THE VISION (Смысл жизни)
Мы делаем "Huntr для бедных, но умных".
Рынок переполнен дорогими ($40/mo) и сложными комбайнами. Наша ниша — скорость и простота.
**Главная ценность:** Юзер должен тратить на трекинг и анализ вакансии 5 секунд, а не 5 минут.
**Слоган:** "Click. Match. Apply. Repeat."
Пользователь все также откликается руками, но инструмент предполагает быстрый отлик и сохранение в базу для последующего анализа.

---

## 2. CORE PRINCIPLES (Понятия)
1.  **No WYSIWYG Hell:** Мы не делаем сложный редактор резюме. Мы делаем *генератор* по шаблону. Для хранения резюме используем json resume формат.
2.  **Extension First:** Главная точка входа — браузерный плагин. Сайт для основной работы, плагин как UI для быстрых отликов, никакого анализа в плагине не делаем и доступ к базе данных не получаем. Минимум UI, пара функциональных кнопок.
3.  **Indie Hacker Stack:** Никаких микросервисов, кубернетисов и оверинжиниринга. Всё должно работать на Free Tier (или близко к нему).
4.  **AI as a Tool, not a Toy:** AI используется только для парсинга (JSON) и сухой аналитики. Написания Cover Letter.

## 3. USER FLOW (Как это работает)

### Создание аккаунта
1. Юзер заходит на сайт, создает аккаунт с помощью любой учетной записи из популярных сервисов (Google, Apple, Facebook, etc). Не делаем сейчас учеткой записи по email и паролю отдельному.
2. Плагин может лишь отображать состояние (залогинен или нет). Если нет, то кнопка логина перенаправляет на сайт для создания / логина. Если пользователь уже залогинился, плагин должен понимать это.

### Создание пустого профиля позиции
Пользователь должен уметь заводить несколько профилей. Профиль из себя представляет логическую сущность (позицию) под которую будут подаваться пользовать.
1. На сайте профиля отображаются влевой панели. По клику на профиль будет открываться окно с откликами.
2. Профилей можно создавать бесконечное число, в реальности скорее всего пользоателю будет достаточно не больше 5.
3. Профили должны отображаться в расширение браузера, и возможность выбирать из выпадающего списка.

### Флоу отлика на вакансию
1.  Юзер находит вакансию (LinkedIn, Indeed, Ashby). На данном этапе мы рассматриваем что пользователь заходит непосредственно на сайт где размещена вакансии.
2.  Открывает расширение и видит там кнопку - Apply.
3.  Плагин грабит текст -> Шлет в LLM API с промптом достать необходимые базовые данные из вакансии -> AI выплевывает JSON (`title`, `company`, `salary`, `stack`, `url`, `jod description`). 
4. В плагине показывается окно с этими полученными полями, чтобы была возможность поправить если LLM отработает неверно.
5. В этом же окно отображается кнопка Save, по нажатию на которую вакансия сохраняется в базу данных откликов под конкретный профиль.

Не надо поддерживать добавление отлика через сайт наш, работаем только через плагин.

### Загрузка резюме в профиль
Пользователь хочет хранить свои резюме в профиле для последующего анализа совпадения через LLM. Редактировать это резюме и создавать кастомизированную версию.
1. Пользователь заходит на сайт, выбирает ранее созданный профиль. 
2. Нажимает кнопку "загрузить резюме"
3. Профиль может хранить несколько версий резюме. Здесь не знаю как лучше реализовать, в общем неплохо иметь версионирование. Потому что в процессе поиска работы и откликов резюме может меняться и появляться новая версия, более свежая и обновленная. Хочется иметь версионирование чтобы потом в статистике смотреть результативность и по версии резюме тоже. Возможно для обновленного конверсия будет больше.

### Работа с откликом
Запись в базе должна содержать несколько полей.
Обязательные поля:
- Company name
- Сфера деятельности компании (fintech, edtech, saas, etc.). Может быть предустановленным списком.
- Поле где хранится описание вакансии в текстовом виде. В таблице наверно нет смысла отображать весь текст. Нужно для последующего анализа совпадения резюме и вакансии
- URL на вакансию
- Country
- CV Version
- Status (single selected field), например HR Screening, 1st Interview, System Desing Interview, Rejected, etc. 
- Date - дата следующего этапа.
- Должна поддерживаться история изменения поля Status. Со временем отклик будет менять свое состояние, переходить из HR screening в rejected. Хочется хранить эту историю, чтобы потом с ней работать и получать информацию.
- Applying date - возможно как отдельное поле не нужно, хочется понимать сколько прошло время между подачей и текущей датой, чтобы руками некоторые вакансии переносить в статус not responded например.
- Rejection reason - текстовое поле, фидбек о работадателя когда отказывают, обычно его нет, но если будет полезен.
- Salary range 
- Notes - текстовое поле для личных заметок

### Другой функционал (который распишу подробнее)

2. Канбан доска для большей наглядности в каком статусе каждый отклик.
